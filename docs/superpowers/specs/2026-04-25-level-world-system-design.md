# §4 Level / World System — Design Spec
**Date:** 2026-04-25
**Status:** Approved, pending implementation plan

---

## Overview

The Level/World System is §4 of the s4_engine architecture. It implements a 2D section grid with bidirectional streaming, pre-computed nametable strips, a deferred plane buffer, a full camera system, per-section collision maps, slot-based entity management with warp-based preview, 8-layer computed parallax, and a data-driven zone database — all without any code or formats ported from sonic_hack.

The system is split into five phases, each independently testable. Every phase has research grounding from all 7 local references and online sources. The first map target is Oracle Jungle Zone (OJZ) using art and layout data migrated from sonic_hack via build tools.

---

## Core Principles

1. **Each section is its own world.** Per-section art, palette, music, collision, objects, rings, parallax, deformation, animated tiles. Zero value = keep current. No section is constrained by another.

2. **Pre-computed at build time, not runtime.** Nametable strips, collision maps, ring patterns, object tables — all generated by the build tool. Runtime never converts chunks or walks lookup chains.

3. **No sonic_hack code or formats survive.** The build tool reads sonic_hack binary files as input and outputs s4_engine formats. All engine code is written fresh.

4. **Slot origins are fixed engine-space positions.** Slot L = `$0200`, Slot R = `$0A00`. Sections are unlimited; only 4 slots exist at runtime. `section_x` does not affect slot_origin — it determines which data loads into the slot.

5. **World coordinates = `(section_x × $800) + local_x`.** Stored per-section in ROM as local coords (0–$7FF); converted to engine space at spawn. `Section_WorldToEngine` / `Section_EngineToWorld` are Phase 1 utility routines.

## Coordinate Space

```
Horizontal engine space:
  [BWD_PREVIEW $000-$1FF][SLOT_L $200-$9FF][SLOT_R $A00-$11FF][FWD_PREVIEW $1200-$13FF]

Vertical engine space (same layout on Y axis):
  [BWD_PREVIEW $000-$1FF][SLOT_U $200-$9FF][SLOT_D $A00-$11FF][FWD_PREVIEW $1200-$13FF]

Teleport thresholds:  fwd=$1200, bwd=$0200
Preload thresholds:   fwd=$0E00, bwd=$0400
SECTION_SHIFT:        $1000

Section size: 16×16 chunks = 2048×2048 pixels
  → 128 blocks per axis (16px/block)
  → ~170 frames to cross at max speed (~12px/frame)
  → ~85 frames lead time from midpoint preload trigger

Sections: unlimited (2D grid, irregular shape supported — null neighbor = camera clamp)
Slots: exactly 4 (LU, RU, LD, RD — addressed as L/R + U/D pairs)
```

## Section Definition Struct — 64 bytes, all phases

```asm
; All fields: 0 = keep current / no change
sec_strips_a:           equ $00   ; .l — pre-computed plane A nametable strips (ROM)
sec_objects:            equ $04   ; .l — compact 4-byte object entry array
sec_rings:              equ $08   ; .l — pattern-encoded ring entry array
sec_plc:                equ $0C   ; .l — S4LZ art PLC list
sec_pal:                equ $10   ; .l — full 128-byte palette (4 lines × 32 bytes)
sec_scroll:             equ $14   ; .l — parallax layer table (Phase 4)
sec_raster_table:       equ $18   ; .l — raster command table (§7.2, future)
sec_strips_b:           equ $1C   ; .l — pre-computed plane B nametable strips (ROM)
sec_reserved:           equ $20   ; .l — reserved (was sec_nametable_b duplicate)
sec_pal_cycle:          equ $24   ; .l — palette cycling script (Phase 4)
sec_sound_bank:         equ $28   ; .l — DAC sample bank pointer
sec_deform_table:       equ $2C   ; .l — deformation table (256 signed bytes, Phase 4)
sec_anim_blocks:        equ $30   ; .l — animated tile script (Phase 4)
sec_collision:          equ $34   ; .l — flat 128×128 byte collision map
sec_flags:              equ $38   ; .w — has_water|underground|no_y_wrap|preserve_state
sec_music:              equ $3A   ; .w — music change (0 = keep current)
sec_layer_mask:         equ $3C   ; .b — parallax layer enable bitmask (Phase 4)
sec_camera_lookahead:   equ $3D   ; .b — camera lookahead pixels (0 = zone default)
sec_deform_speed:       equ $3E   ; .b — deformation rate (1=normal, Phase 4)
sec_transition_type:    equ $3F   ; .b — 0=blend, 1=instant, 2=fade (Phase 4)
sec_entry_size:         equ $40
```

## Phase 1 — Core Scroll Infrastructure + Section Teleport

**Delivers:** Visible OJZ level scrolling, 2D bidirectional section teleport, camera, minimal parallax.
**Unlocks:** §1 deferred scroll work (Draw_TileColumn/Row, VInt_DrawLevel, HScroll/VScroll DMA).

### Research Grounding

- **sonic_hack `section_streaming.asm`** (1388 lines) — validates 2D leapfrog + teleport mechanics directly. Teleport thresholds, preview zones, slot structure, coordinate constants all match.
- **Batman & Robin** — 909KB of raw VDP nametable data in ROM, zero CPU at scroll time. Directly validates pre-computed strip approach.
- **S.C.E. `Draw Level.asm` + `Plane_buffer`** — validates producer-consumer deferred plane write. `VInt_DrawLevel` pattern, `Plane_double_update_flag` for fast scroll.
- **Online: Kabuto hardware notes** — VDP autoincrement `$8F80` (stride=$80) is the key trick for column writes: each sequential word write drops one row in the nametable. Set before column DMA, restore to `$8F02` after.
- **Online: md.railgun.works** — 64×64 nametable costs 8KB per plane (vs 4KB for 64×32). Both planes = 16KB total overhead, leaving ~46KB for art pool. Worth it: 64×32 gives only 32px vertical off-screen buffer; 64×64 gives 288px — essential for vertical streaming.
- **Online: plutiedev DMA notes** — per-8-row HScroll costs 112 bytes vs 896 for per-scanline. 784 bytes of saved VBlank DMA budget for tile streaming. Sufficient for fixed-ratio BG parallax in Phase 1.

### Plane Configuration

- **64×64 scroll planes** (both A and B). 8KB per plane, 16KB total for nametables. ~46KB remaining for art pool.
- VDP register 16 = `$01` (64×64). Set during boot, never changed.
- VDP register 11 = `$02` (per-8-row HScroll) for Phase 1. Upgraded to `$03` (per-scanline) in Phase 4.

### Nametable Strips + Build Tool

- `tools/ojz_strip_gen.py` — reads sonic_hack OJZ binary files (chunk layout, block mappings, tile data), outputs pre-computed nametable strip arrays as `dc.w` ROM includes. Handles X/Y-flipped tiles, palette bits, priority bits.
- One strip = one 16-tile-wide column of nametable words for one section. 16 strips per section = one full section width.
- Scroll engine never converts chunks at runtime. New column = DMA one strip from ROM to the off-screen VDP plane columns.
- **Dynamic override table** — 16 entries per section in RAM: `(column_index, tile_offset, new_tile_word)`. `Draw_TileColumn` checks table before DMA and patches overridden tiles. Handles breakable floors, mutable terrain. Cleared on section unload.
- **Editor workflow:** Editor (SonLVL for now) saves per-section layout files in standard binary format (16×16 words = 512 bytes). Build tool reads these as input. Editor never knows strips exist.
- **Future editor format:** one binary layout file (512 bytes, 16×16 word chunk indices) + one TOML sidecar per section for metadata (palette, music, flags, etc.). Build tool reads both; outputs generated ROM data to `data/generated/` (gitignored).
- **File organisation:** `data/levels/ojz/act1/sec_0_0.bin`, `sec_0_0.toml`, `sec_0_1.bin`, etc. One file pair per section cell.

### Deferred Plane Buffer

- `Plane_Buffer`: 768 words (1536 bytes) in RAM. Game loop appends entries; VBlank drains.
- Entry format: `[VRAM_addr.l][word_count.w][tile_words...]`
- `Draw_TileColumn(col, section_ptr)` — appends one column's strip to buffer. Sets VDP autoincrement `$8F80` in the entry header so VBlank drains with correct vertical stride.
- `Draw_TileRow(row, section_ptr)` — appends one row to buffer with `$8F02` stride.
- **Double-update:** if camera moved >16px in one frame, `Draw_TileColumn` called twice for that axis.
- `VInt_DrawLevel` — called from `vblank.asm` after DMA queue drain. Processes entries sequentially, restores increment to `$8F02` after each column entry. Overflow check before append: if full, defer to next frame.
- Both Plane A and Plane B updated: `sec_strips_a` and `sec_strips_b` each generate their own buffer entries.

### Section Teleport Engine

- `Section_Init(act_ptr)` — sets up initial 2×2 slot grid from act descriptor. DMA-strips the full visible area to VDP. Establishes slot origins and `Slot_Section_Map`.
- `Section_Check` — per-frame horizontal: checks camera X against preload (`$0E00`) and teleport (`$1200`/`$0200`) thresholds. Fires preload or teleport as needed.
- `Section_CheckVertical` — per-frame vertical: same logic on Y axis.
- `Section_Teleport(direction)` — applies uniform `±SECTION_SHIFT` to camera and all `Slot_Origins`. Swaps slot roles (L↔R or U↔D). Updates `Slot_Section_Map`. Queues art/palette for incoming section. Total CPU on teleport frame: ~1,500 cycles.
- `Section_Preload(direction)` — queues S4LZ art streams as Priority 2 (deferrable) DMA. Sets preload flag. Velocity-based timing: `preload_at = threshold - (abs(vel) × lead_frames)` so fast players get earlier preloads.
- **Diagonal corners:** both axes fire independently. Diagonal preload is Priority 2 (deferrable). No special-casing — corner becomes a normal neighbour after first teleport.
- **Null neighbour:** `Level_GetSection` returns null → camera clamp. Boundary is data-driven; no hardcoded level extents.

### Camera

- `Camera_X` / `Camera_Y`: 16.16 fixed-point (already in `ram.asm`).
- **X deadzone:** ±16px from screen centre. Velocity-adaptive: `deadzone = $10 + (abs(x_vel) >> 3)`. Widens at top speed to show more ahead.
- **Y tracking:** `Distance_from_top = $60` (96px from top). ±32px airborne deadzone. Speed-dependent scroll rate.
- **Extended lookahead (S.C.E.-validated):** pan ±64px at ≥`$600` ground speed, 2px/frame rate. `Camera_Pan_Offset` tracks current pan.
- **Position history buffer:** `Pos_table` — 64 entries × 4 bytes (256 bytes). Stores last 64 frames of player X/Y. `H_scroll_frame_offset` selects lag depth. Whip-effect follow at top speed.
- **Bounds:** clamped by act descriptor `cam_min/max_x/y`. Section null-neighbour check prevents camera reaching unmapped space.

### HScroll (Phase 1 Minimal)

- Per-8-row mode: 14 entries × 8 bytes (FG + BG word pair) = 112 bytes. Saves 784 bytes VBlank DMA for tile streaming.
- `Hscroll_Update`: Plane A = `camera_x`, Plane B = `camera_x >> 1`. DMA'd as single transfer in VBlank.

### World Coordinate Utilities

```asm
; Section_EngineToWorld — engine_x + slot state → world_x (for checkpoint save)
; In:  d0.w = engine_x, d1.b = slot_index
; Out: d0.l = world_x = (slot_section_x × $800) + (engine_x - slot_origin)

; Section_WorldToEngine — section_x + local_x → engine_x (for checkpoint respawn)
; In:  d0.b = section_x, d1.w = local_x
; Out: d0.w = engine_x = $0200 + local_x  (Section_Init places target section in Slot L)
```

These live in the section streaming file. Used by checkpoint objects (§5) and SRAM saves (§9.6).

## Phase 2 — Collision System

**Delivers:** Sonic can stand on OJZ terrain. Floor/wall/ceiling sensors operational.
**Unlocks:** §5 Player physics.

### Research Grounding

- **sonic_hack `object_touch.asm`** — `FindFloor`, `FindWall` routines. Standard Sonic collision chain: tile→block collision ID→heightmap→angle. `ColArray` (16 bytes per shape), `ColCurveMap` (angle per shape), `ColArrayRot` for wall/ceiling.
- **S.C.E.** — same system, inherited from S3K. Confirms dual-pointer primary/secondary collision.
- **Online: Sonic Community Hacking Guide** — `lsl #4` replaces all multiply in height lookup (~40-60 cycles per sensor). `ColArrayRot` at `+$1000` offset eliminates coordinate rotation for wall/ceiling sensors.
- **ENGINE_ARCHITECTURE.md §4.7 correction:** stated "2-4KB per section (128 columns × 16-32 rows)". Correct figure: 128×128 = **16KB per section**. A 2048px section / 16px per block = 128 blocks per axis. For OJZ Act 1 (4×3 = 12 sections): 192KB. Under 5% of a 4MB ROM.

### Flat Per-Section Collision Map

128×128 byte array per section — one collision type ID per 16×16 block. Pre-computed by build tool. Runtime lookup:

```asm
; ~20 cycles, zero multiply
; a0 = section collision map base (from sec_collision)
; d0 = section-local X, d1 = section-local Y
    lsr.w   #4, d0          ; X → block column (0-127)
    lsr.w   #4, d1          ; Y → block row (0-127)
    lsl.w   #7, d1          ; row × 128
    add.w   d0, d1
    move.b  (a0,d1.w), d0   ; collision type byte
```

### Global Height + Angle Tables (shared across all sections)

- `ColArray`: 256 shapes × 16 bytes = 4KB — height per x-pixel within a 16px block
- `ColArrayRot`: 4KB — same data for wall/ceiling sensors (eliminates coordinate rotation)
- `ColCurveMap`: 256 bytes — terrain angle per collision shape (0–$FF = full rotation)
- Ported from OJZ data by build tool. One set globally; the flat map determines which shape appears where per-section.

### Two-Layer Collision

Primary (foreground) and secondary (background) map per section. Secondary checked only when primary returns passable — allows ledge pass-through from below. `Collision_Map_Ptr` array holds both pointers per slot.

### Sensor Routines

- `Find_Floor(x, y)` → distance to ground + terrain angle. Two sensors (left/right foot at ±width/2).
- `Find_Wall(x, y)` → wall distance. Two sensors (left/right side).
- `Find_Ceiling(x, y)` → ceiling distance. Two sensors.
- Height lookup: `ColArray[(collision_type × 16) + (x & $F)]` — single indexed read.
- X-flip: invert x index (`not.w, and.w #$F`). Y-flip: negate height, adjust angle +`$40`.

### Build Tool Extension

`ojz_strip_gen.py` gains collision pass:
- Reads OJZ block→collision_index binaries (primary + secondary)
- Reads chunk→block mappings
- For each section, resolves all 128×128 block positions to collision type IDs
- Outputs per-section flat byte arrays as ROM includes
- Extracts `ColArray`, `ColArrayRot`, `ColCurveMap` from OJZ data as shared ROM includes

| Symbol | Size | Purpose |
|---|---|---|
| `Collision_Map_Ptr` | 32 B | 4 slots × (primary.l + secondary.l) |

## Phase 3 — Entity Management (Rings, Objects, State)

**Delivers:** Rings collectible, objects spawning/despawning with sections, backtrack state preserved.
**Unlocks:** §3 deferred DPLC lookahead, §5 Player system (checkpoint respawn).

### Research Grounding

- **Thunder Force IV** — type-segregated pools validate separate ring/object processing. Rings: stateless, high-volume, fast path. Objects: stateful, lower-volume, SST-based.
- **sonic_hack `section_streaming.asm`** — object slot tagging and despawn-by-slot pattern validated.
- **S.C.E.** — flat array + viewport culling requires global rebuild on section change. Slot-based lifecycle eliminates this entirely.
- **No commercial or homebrew Genesis game** implements warp-based entity preview. Original design.
- **Pattern encoding tested against OJZ Act 1** (240 rings): ~58% ROM savings over raw X/Y pairs.

### Ring Layout — Pattern-Encoded ROM Storage

Three entry types, 4 bytes each:

```
%00 individual:      [type:2][X:11][Y:11][reserved:8]
%01 horizontal line: [type:2][X:11][Y:11][count:5][spacing:3]
%10 vertical line:   [type:2][X:11][Y:11][count:5][spacing:3]

Spacing index: 0=$10, 1=$14, 2=$18, 3=$1C, 4=$20, 5=$24, 6=$28, 7=$30
```

At section load, patterns expand into the slot's ring buffer (section-local X/Y word pairs, 4 bytes/ring). 768 bytes shared across 4 slots = 192 bytes per slot = 48 rings per slot. Expansion is one-time (~50 cycles/pattern). Per-frame: `screen_x = slot_origin_x + ring.local_x - camera_x`.

### Object Layout — Compact 4-Byte ROM Entries

```
[11-bit X][11-bit Y][5-bit type][5-bit subtype] = 32 bits

; X, Y: section-local (0-$7FF)
; type: index into per-section type table (0-31)
; subtype: object-specific parameter (0-31)
```

Per-section type table: up to 32 entries × 4 bytes (routine pointers) = 128-byte ROM array. At section load, copies to `Object_Type_Table` RAM. Spawning: indexed `move.l` for routine pointer → `AllocSlot`.

Objects tag SST scratch byte with slot index on spawn. `Section_UnloadSlot` sweeps Object_RAM, calls `FreeSlot` on all objects matching the slot tag.

### Warp-Based Preview

On section preload: entities spawn at `engine_x = slot_origin + SECTION_SHIFT + local_x` — live in preview zone ($1200+), fully active. Rings: `slot_origin` set to `normal_origin + SECTION_SHIFT` — one word write, all rings in preview zone.

On teleport: uniform `-SECTION_SHIFT` to all positions (camera, slot origins, all SST x_pos/y_pos). Distances between entities preserved (spatial invariant). Rings: set `slot_origin = normal_origin` — one word write, all rings unwarp.

### Rolling 4-Slot State Buffer

```asm
; 4 entries × 26 bytes = 104 bytes total
rolling_state:
    section_id:   ds.b 1     ; $FF = empty
                  ds.b 1     ; pad
    ring_mask:    ds.b 16    ; 128 rings max (1 bit each)
    obj_mask:     ds.b 8     ; 64 objects max (1 bit each)
```

On section load: search buffer for `section_id`, apply bitmasks (collected rings stay gone, dead enemies stay dead). On unload: save bitmasks, evict oldest (circular). Max backtrack before respawn: ~$800px. Sections with `preserve_state` flag write to permanent table instead — survives indefinitely, SRAM-snapshotted.

### Checkpoint Integration (implemented in §5, noted here)

Checkpoint object stores `(section_x, section_y, local_x, local_y)` + rolling state snapshot at touch time. On death/respawn: `Section_Init` with checkpoint section coords, Sonic at `$0200 + local_x`, state buffer restored from snapshot. Uses `Section_WorldToEngine` from Phase 1.

### Build Tool Additions

- Ring pass: reads OJZ raw ring X/Y pairs, detects horizontal/vertical runs, outputs pattern-encoded arrays per section.
- Object pass: reads OJZ object layout (6-byte sonic_hack entries), outputs compact 4-byte entries + per-section type table arrays.

## Phase 4 — Parallax + Animated Tiles

**Delivers:** Full 8-layer computed parallax with per-section deformation, animated terrain tiles.
**Unlocks:** Full visual fidelity. Activates `sec_scroll`, `sec_deform_table`, `sec_layer_mask`, `sec_deform_speed`, `sec_anim_blocks` section fields.
**Note:** Full raster command table (§7.2) — palette swaps per scanline, shadow/highlight toggling, mid-frame VDP register changes — is deferred to the §7 Visual Effects System phase. Phase 4 establishes the HBlank infrastructure §7.2 plugs into.

### Research Grounding

- **Thunder Force IV** — 8 active parallax layers, computed formula `factor = -(layer_index × 2 - 7)`, deformation table at ROM `$17A00` (256 signed bytes, indexed by rolling frame counter). Per-layer scroll directly validated.
- **S.C.E. `HScroll_Deform`** — velocity-accumulator per scanline (16.8 fixed-point), full table built in RAM during game loop, DMA'd as single 896-byte transfer in VBlank. More bandwidth-efficient than H-interrupt per scanline.
- **S.C.E.** — dual FG/BG deformation state (independent tables, accumulators, speeds). Per-block linear interpolation flag (high bit controls smooth gradient vs hard edge per 8-scanline block).
- **Online: plutiedev** — HScroll per-8-row (Phase 1) → per-scanline (Phase 4) upgrade: VDP register 11 `$02` → `$03`. Only change needed to enable full per-line deformation.
- **Batman & Robin** — `sec_raster_table` per-section pointer validated. Phase 4 activates this field for scroll purposes; §7.2 adds full multi-effect dispatch.

### 8-Layer Computed Parallax

Phase 4 upgrades `Hscroll_Update` from the Phase 1 fixed-ratio to full computation. VDP register 11 changes from `$02` (per-8-row) to `$03` (per-scanline). Hscroll_Buffer already allocated at 896 bytes in `ram.asm`.

```asm
; Per layer during game loop (~140 cycles total for 8 layers):
; factor = -(layer_index × 2 - 7)
; deform = deform_table[(frame_counter + layer_offset) & $FF] × deform_speed
; layer_scroll = (camera_x × factor >> shift) + deform_accumulator
; → written into Hscroll_Buffer for all scanlines this layer covers
```

`Hscroll_Dirty_Start`/`Hscroll_Dirty_End` from `ram.asm` limits the VBlank DMA to only changed scanlines.

**Per-section layer config (sec_scroll):** 8-entry layer table — scroll factor, phase offset, initial scroll value. Each section can have completely different parallax behaviour.

**Layer enable mask (sec_layer_mask):** Disables unused layers per section. Underground sections skip sky layers — saves ~140 cycles/layer/frame and reduces DMA load.

**Parallax transition smoothing (NOVEL):** At section boundaries with different scroll config: `current += (target - current) >> 3` per layer per frame over 8-16 frames. `sec_transition_type = 1` forces instant swap. No Genesis game smoothly transitions parallax mid-level.

**Dual FG/BG deformation:** Separate deformation state for Plane A and Plane B — independent `sec_deform_table` lookups, accumulators, and `sec_deform_speed` values. Foreground and background wave independently.

**Per-block linear interpolation:** High bit of each 8-scanline deformation block selects smooth gradient (interpolate offsets across block) vs hard edge (repeat raw value). Allows mixing smooth water ripples with sharp terrain boundaries in one table.

### Animated Tiles Per-Section

`sec_anim_blocks` points to an animation script:

```asm
; Animation script entry (per animated tile set in this section):
;   dc.l  vram_dest        ; VRAM tile destination
;   dc.w  frame_count      ; number of frames in cycle
;   dc.w  frame_period     ; VBlank ticks per frame
;   dc.l  frame_ptrs[N]    ; pointers to uncompressed tile data
```

Each VBlank: advance frame counter, DMA current frame's tile data to `vram_dest` via Priority 1 (important) DMA queue. On section load: start animation at frame 0. On section unload: stop (VRAM overwritten by next section's art load).

Use cases: conveyor belts, pulsing lava, swaying grass, shimmering water surface, flickering torches — each section gets its own independently defined animated terrain.

### VScroll Per-Column

If `sec_flags` has column-scroll bit set: write per-column VSRAM entries (20 entries × 4 bytes = 80 bytes) directly during game loop. VSRAM writes are fast (no FIFO stall). Enables tilting foreground planes and section-level tilt effects.

## Phase 5 — Level Database + Build Pipeline

**Delivers:** Data-driven zone→act→section lookup. New zones require zero engine code changes.
**Unlocks:** Multiple zones, full build tool generalisation.

### Research Grounding

- **sonic_hack `level_database.asm`** (70 lines) + `zones/ojz.asm` (205 lines) — validate the zone→act→section lookup chain directly. Phase 5 mirrors this structure with the richer 64-byte section definition.
- **S.C.E. `Level_data_addr_RAM`** (162-byte struct) — validates the per-act descriptor approach.
- **No commercial or homebrew game** has a fully data-driven zone database requiring zero engine code per new zone. Clean engineering improvement over all references.

### Zone→Act→Section Database

```asm
LevelDatabase:
    dc.l    ZoneDB_OJZ      ; zone 0
    ; new zones: add dc.l here only

ZoneDB_OJZ:
    dc.l    OJZ_TileArt     ; shared zone tile art (S4LZ)
    dc.l    OJZ_Palette     ; default palette
    dc.w    ZoneDB_OJZ_Acts-ZoneDB_OJZ

ZoneDB_OJZ_Acts:
    dc.w    OJZ_Act1-ZoneDB_OJZ
    dc.w    OJZ_Act2-ZoneDB_OJZ

OJZ_Act1:                           ; act descriptor
    dc.l    OJZ_Act1_Sections       ; section grid pointer
    dc.w    grid_w, grid_h          ; 4, 3
    dc.w    start_local_x, start_local_y
    dc.b    start_section_x, start_section_y
    dc.w    cam_min_x, cam_max_x
    dc.w    cam_min_y, cam_max_y

OJZ_Act1_Sections:
    ; 4×3 = 12 section definitions, each sec_entry_size ($40) bytes
```

### Lookup Routines

- `Level_GetZone(d0.b)` → `a0` = zone descriptor. ~8 instructions.
- `Level_GetAct(a0, d1.b)` → `a1` = act descriptor. ~8 instructions.
- `Level_GetSection(a1, d0.b section_x, d1.b section_y)` → `a0` = section definition. ~10 instructions.
- All pure indexed arithmetic — no branching, no searching.

### Full Level Init Sequence

```
Level_Init(zone_id, act_id):
  1. Level_GetZone → queue shared zone art (S4LZ, Priority 2)
  2. Level_GetAct  → load act descriptor (bounds, start, grid dims)
  3. Section_Init  → build 2×2 slot grid, DMA visible strips to VDP
  4. Camera_Init   → set bounds, start position, history buffer clear
  5. Collision_Init→ set Collision_Map_Ptr for starting sections
  6. Entity_Init   → expand ring patterns, spawn starting objects
  7. Palette_Load  → load starting section palette to Palette_Buffer
  8. Audio_Init    → start section music
```

### Build Tool Generalisation

Phases 1–4 use `ojz_strip_gen.py` (OJZ-specific). Phase 5 refactors into:

```
tools/
  level_build.py      ; master entry: level_build.py <zone> <act>
  strip_gen.py        ; chunk layout → nametable strips (any zone)
  collision_gen.py    ; chunk collision → flat byte arrays (any zone)
  ring_encode.py      ; raw ring X/Y pairs → pattern-encoded (any zone)
  object_encode.py    ; object layout → compact 4-byte + type tables
  section_meta.py     ; TOML sidecar → 64-byte section struct include
```

`level_build.py <zone> <act>` discovers the zone's `data/levels/<zone>/act<n>/` directory, runs all passes, writes `data/generated/<zone>/act<n>/` includes. Adding a new zone = new source directory + TOML sidecars. No engine code touched.

## Key Invariants

1. **Slot origins are fixed:** Slot L = `$0200`, Slot R = `$0A00` always. `section_x` index selects what data is in the slot, not where the slot lives.
2. **SECTION_SHIFT = `$1000`** — the preview zone is exactly one slot-width ahead. All entities in preview zone are real, fully active.
3. **Teleport is a uniform shift:** `-SECTION_SHIFT` applied to camera + all slot origins. Distances between all entities are preserved. No special cases.
4. **Zero value = keep current** in all section definition fields. A minimal section needs only a layout pointer and collision pointer.
5. **Build tool owns all indirection.** Runtime never walks chunk→block→tile chains or collision index chains. One lookup per system, pre-computed.
6. **Rings never store engine-space positions.** Slot origin handles all coordinate translation. Warp/unwarp = one word write to `Slot_Origins`.
